# NEXORA AI — نظام تتبع فتح وضغط الإيميل (Open & Click Tracking)

هذا المجلد يحتوي على النظام كامل: سيرفر + قاعدة بيانات JSON + لوحة تحكم (Dashboard) + قالب الإيميل مربوط بالتتبع.

## 📁 محتويات المجلد
- `server.js` — السيرفر (Node.js / Express)
- `package.json` — الحزم المطلوبة
- `dashboard.html` — لوحة التحكم (تعرض الأرقام الحقيقية فقط، بدون أي بيانات وهمية)
- `NEXORA_AI_Email_v7.html` — قالب الإيميل، مربوط الآن بالبكسل وزر الـ CTA
- `data/events.json` — قاعدة بيانات الأحداث، مصفّرة `[]` جاهزة للبيانات الحقيقية

---

## 1) التشغيل محلياً (اختياري، للتجربة فقط)
```bash
npm install
npm start
```
السيرفر سيعمل على `http://localhost:3000` واللوحة على نفس الرابط `/`.

## 2) النشر (Deploy) — خطوات حقيقية
1. ارفع هذا المجلد بالكامل كـ repo على GitHub.
2. اذهب إلى railway.app (أو Render / Fly.io) → New Project → Deploy from GitHub → اختر الـ repo.
3. بعد النشر ستحصل على رابط مثل: `yourid.up.railway.app`
4. هذا الرابط هو الـ **SERVER_URL** الذي سنستخدمه في الخطوة التالية.

⚠️ مهم: بيانات `data/events.json` تُحفظ على القرص المحلي للسيرفر. على Railway/Render المجاني القرص **غير دائم** (ephemeral) — أي إعادة نشر (redeploy) قد تمسح البيانات القديمة. إذا احتجت أرشفة دائمة للأرقام على المدى الطويل، أخبرني لأضيف تخزين خارجي (قاعدة بيانات حقيقية).

## 3) ربط الإيميل بالسيرفر (لكل إيميل يُرسل)
قالب `NEXORA_AI_Email_v7.html` يحتوي الآن على مكانين تم استبدالهما:

- `{{SERVER_URL}}` → استبدله برابط السيرفر الفعلي (بدون https://، فقط الدومين، مثال: `yourid.up.railway.app`)
- `{{EmailID}}` → **يجب أن يكون فريدًا لكل متلقٍ** (مثلاً بريد العميل نفسه أو رقم تسلسلي)، لأن هذا هو المعرّف الذي تُبنى عليه كل الإحصائيات (Unique Opens / Unique Clicks لكل عميل).

هذان الاستبدالان موجودان في 3 مواضع بالملف:
1. بكسل الفتح (قبل إغلاق الإيميل مباشرة)
2. زر الـ CTA بصيغة Outlook (VML)
3. زر الـ CTA بصيغة HTML العادية

إذا كنت ترسل عبر أداة Mail Merge (Google Sheets + Apps Script، أو Mailchimp، أو غيره)، اجعل الأداة تستبدل `{{EmailID}}` تلقائيًا بعنوان بريد كل متلقٍ قبل الإرسال.

## 4) لوحة التحكم
- تبدأ من الصفر فعليًا (0 فتحات، 0 ضغطات) — لا توجد أي أرقام أو بيانات تجريبية/عشوائية.
- إذا تعذّر الوصول للسيرفر، تظهر رسالة واضحة "Server unreachable" بدل عرض أرقام وهمية.
- بمجرد نشر السيرفر وإرسال أول إيميل حقيقي، الأرقام تُحدَّث تلقائيًا كل 30 ثانية.
- زر "Refresh" يجلب البيانات فورًا من `/api/stats`.

## 5) ملاحظات مهمة على دقة الأرقام (مصداقية التحليل)
- **Apple Mail Privacy Protection**: أجهزة آبل (iPhone/iPad/Mac Mail) تُحمّل صورة البكسل تلقائيًا لكل الإيميلات حتى لو المستخدم لم يفتحه فعليًا — هذا يُضخّم رقم "Opens" وهي مشكلة معروفة في كل أنظمة تتبع البريد وليست خطأ في هذا النظام. لذلك **اعتمد على "Clicks" كمؤشر أدق على الاهتمام الحقيقي** وليس فقط "Opens".
- Gmail يُمرر الصور عبر سيرفراته (proxy) — هذا طبيعي ولا يؤثر سلبًا على دقة رقم الفتح نفسه.
- تأكد أن `{{EmailID}}` مختلف لكل متلقٍ، وإلا ستُحسب كل الفتحات/الضغطات كأنها من شخص واحد فقط.
- رابط الضغط (`/click`) الآن يتحقق من أن الرابط الموجَّه إليه `http/https` صالح فقط (حماية من إساءة استخدام الرابط كـ Open Redirect).

## 6) نقاط الوصول (Endpoints)
| Endpoint | الوظيفة |
|---|---|
| `GET /pixel?id=X` | يسجّل "فتح" ويرجّع صورة شفافة 1×1 |
| `GET /click?id=X&url=Y` | يسجّل "ضغطة" ثم يحوّل المستخدم لـ Y |
| `GET /api/stats` | يرجّع الإحصائيات المُجمّعة (تستخدمها اللوحة) |
| `GET /api/events?id=X` | يرجّع كل الأحداث، أو أحداث بريد معيّن |
| `GET /api/health` | فحص سريع أن السيرفر شغال |
